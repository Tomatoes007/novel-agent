Novel Agent 项目功能与迭代说明文档

一、项目概述

1. 项目定位
本项目是一个面向中文网络小说创作场景的 AI Agent 后端系统，目标是帮助用户完成从小说立项、世界观构建、角色设计、大纲生成，到章节写作、记忆维护、检索增强和内容审校的一整套创作辅助流程。

该项目当前重点不是“直接生成一整本小说”，而是构建一套适合长篇连载写作的后端工作流系统。系统希望解决以下问题：
1）作者起书困难，不知道如何快速完成立项与设定；
2）长篇写作中容易出现角色崩坏、设定冲突、时间线混乱；
3）普通聊天式大模型缺少长期记忆，无法稳定支持多章续写；
4）网文创作需要强调节奏、爽点、钩子，而不是单纯文学化表达；
5）需要把“项目管理 + Prompt + Tool + Memory + 检索增强”整合成一个真正可运行的 AI 应用后端。

2. 技术架构定位
当前项目采用分层式后端架构，核心技术栈如下：
- FastAPI：提供 API 服务与 Swagger 文档；
- SQLAlchemy：负责 ORM 与结构化数据持久化；
- MySQL：存储项目、章节、角色、设定、大纲、伏笔、时间线等结构化数据；
- Redis：缓存最近章节摘要、上下文快照等热点数据；
- Milvus：存储向量化记忆，用于章节生成时的语义检索；
- OpenAI 兼容 SDK：用于对接千问或其他兼容接口模型；
- Pydantic：定义请求体、响应体与配置项。

3. 当前项目特点
当前版本属于“可启动、可扩展、具备核心主链路雏形”的后端骨架版本，已经具备：
- 启动服务；
- 建表；
- 管理项目基础信息；
- 调用大模型生成部分内容；
- 维护部分结构化记忆；
- 预留检索增强接口与多模块扩展空间。

但同时它还不属于“完全成熟可商用版本”，仍有不少功能需要继续补充和强化。


二、当前项目已实现内容

1. 基础工程结构已完成
当前项目已经具备完整的后端工程目录划分，主要包括：
- core：配置、数据库连接、Redis 客户端、Milvus 客户端、LLM 客户端；
- models：项目、角色、世界观、章节、大纲、伏笔、时间线、记忆块等数据库模型；
- schemas：请求与响应的数据结构定义；
- repositories：数据库访问层；
- services：业务逻辑层；
- agents：Agent 编排、上下文加载、章节生成等核心流程；
- prompts：Prompt 模板定义；
- api：FastAPI 路由定义。

这意味着项目已经具备较清晰的分层结构，不是单文件式 Demo，而是按照可扩展项目的方式组织。

2. FastAPI 服务可正常启动
当前项目可以正常通过 uvicorn 启动，并且根路由 / 与健康检查路由 /health 可以正常返回结果。这表明：
- 项目环境安装基本正确；
- 配置加载逻辑正确；
- 数据库初始化逻辑可以执行；
- FastAPI 主应用和路由挂载正常。

3. 配置系统已完成
项目已经实现基于 .env 的配置读取能力，主要支持以下配置：
- MySQL 连接参数；
- Redis 连接参数；
- Milvus 连接参数；
- 大模型接口参数；
- Embedding 模型与向量维度参数。

这意味着系统支持后续根据不同环境进行切换，例如本地开发环境、测试环境、线上环境。

4. MySQL 数据模型已实现
当前项目已经设计并实现了主要的数据库表结构，包括：
- projects：项目基础信息；
- world_settings：世界观与规则；
- characters：角色卡及动态状态；
- outlines：总纲、卷纲、章节纲；
- chapters：章节正文与摘要；
- foreshadowings：伏笔信息；
- timeline_events：时间线事件；
- memory_chunks：结构化记忆片段。

这些模型已经足以支撑小说项目的基本业务流程。

5. 项目管理接口已实现
当前项目已经实现基础项目管理能力，包括：
- 创建项目；
- 获取项目列表；
- 获取单个项目信息；
- 更新项目信息（视具体代码版本而定，部分可能已预留）。

这一部分可用于后续多项目管理。

6. Proposal（立项方案）生成接口已实现
当前项目已经实现：
- POST /api/v1/agent/proposal

该接口可以根据用户输入的创意描述，调用大模型生成初步立项方案，例如：
- 书名备选；
- 核心卖点；
- 题材方向；
- 主角设定建议；
- 前期吸引点建议。

这一能力说明 LLM 调用主链路已经通了，只要配置正确，就可以返回生成结果。

7. 世界观生成接口已实现
当前项目已经实现世界观生成的基础能力。大致流程为：
- 读取项目基础信息；
- 调用模型生成世界观内容；
- 写入 MySQL；
- 提供查询接口。

该模块已经能承担“项目设定初始化”的作用。

8. 角色生成接口已实现
当前项目已实现角色生成能力。大致流程为：
- 根据项目设定调用角色生成 Prompt；
- 生成多名角色的结构化信息；
- 保存角色卡到数据库；
- 支持按项目查询角色列表。

该能力已经可以支撑后续章节生成时读取角色上下文。

9. 大纲生成接口已实现
当前项目已实现大纲生成功能。现阶段主要包括：
- 根据项目、世界观、角色信息生成卷纲；
- 根据卷目标生成若干章节纲；
- 保存到 outlines 表中。

这意味着项目已经不仅是“自由生成文本”，而是具备“先规划、再写作”的结构化流程。

10. 章节生成主链路已实现
这是当前项目最核心的已实现能力之一。

已实现流程：
1）读取项目基础信息；
2）读取当前卷纲/章节纲；
3）读取角色卡；
4）读取世界观规则；
5）读取最近章节摘要；
6）读取开放伏笔；
7）调用检索服务，尝试从 Milvus 中取回相关记忆；
8）拼装 Prompt；
9）调用大模型生成章节；
10）解析结构化输出；
11）保存章节正文与摘要；
12）写回部分记忆。

这一链路已经体现出 Agent 的核心特征：
不是简单聊天，而是先加载上下文，再生成内容，再进行状态更新。

11. 章节摘要与记忆更新能力已实现
当前项目在章节生成后，已经支持把生成结果中的部分信息写回记忆系统，主要包括：
- chapter_summary；
- new_foreshadowing；
- resolved_foreshadowing；
- timeline_events；
- power_changes（部分角色状态变化）。

这说明系统已经开始具备“写完一章之后记住发生了什么”的能力。

12. Redis 缓存能力已接入
当前项目中 Redis 已接入，并封装了部分缓存服务，当前主要用于：
- 最近章节摘要缓存；
- 项目上下文快照缓存（部分为预留能力）；
- 后续还可扩展至幂等控制、接口限流等。

虽然目前 Redis 使用还不算重，但基础接入已经完成。

13. Milvus 检索能力已接入
当前项目已经接入 Milvus，并已实现：
- collection 初始化；
- embedding 后写入向量；
- 根据 query 进行语义检索。

当前设计中，Milvus 主要承担“长篇写作语义记忆召回”的角色，例如：
- 找回与当前章节相关的历史摘要；
- 找回某个伏笔相关的旧片段；
- 找回某角色的重要历史事件。

虽然当前检索策略仍较基础，但基础设施已打通。

14. Swagger 文档已可用
当前项目可以通过 /docs 查看接口文档，并在页面中直接测试接口，这大大降低了联调门槛。


三、当前项目已经具备的可测试能力

从“实际可测试”的角度，目前项目已经可以测试以下内容：

1. 服务启动
可测试服务是否成功启动，并访问：
- /
- /health

2. 创建项目
可通过接口创建小说项目。

3. 获取项目信息
可获取项目列表和详情。

4. 立项方案生成
可提交用户创意，测试千问/兼容模型是否正常返回 proposal。

5. 世界观生成
可基于项目信息生成世界观并入库。

6. 角色生成
可基于项目生成角色卡并入库。

7. 大纲生成
可生成卷纲和章节纲。

8. 章节生成
在前置数据存在的前提下，可以测试第 N 章生成。

9. 章节查询
可查询已生成章节。

10. 记忆相关查询
包括：
- 伏笔列表；
- 时间线列表；
- 最近摘要；
- 向量检索结果（前提是 embedding 与 Milvus 正常）。

11. 审校功能（若路由已接入）
可对章节进行评审和简单分析。


四、当前项目存在的不足与限制

1. 大模型错误处理还不够完善
当前项目在大模型调用失败时，主要表现为直接抛异常，例如：
- API key 错误；
- 鉴权失败；
- 网络失败；
- 模型名错误；
- embedding 请求失败。

目前缺少更友好的异常包装和统一错误返回格式。

2. JSON 输出解析仍然偏脆弱
当前章节生成、角色生成、大纲生成等场景，很多地方依赖模型返回结构化 JSON。
如果模型输出不规范，当前解析可能失败，或者需要依赖正则抽取。

这说明后续还需要：
- 更稳定的结构化输出约束；
- schema 校验；
- 失败重试；
- 降级逻辑。

3. 检索增强策略较基础
当前虽然已经接入 Milvus，但检索策略还比较初步，存在以下问题：
- chunk 粒度还不够精细；
- query 构造还较简单；
- 召回结果还没有做更细的 rerank；
- 检索回来的内容没有强约束进入生成链路。

因此当前的检索增强更多是“雏形”，还不是成熟 RAG 系统。

4. 角色动态状态更新还不完善
当前项目虽然支持写回部分 power_changes，但整体角色状态机还不够丰富。
例如以下状态还缺少系统性维护：
- 当前地点；
- 当前情绪；
- 当前关系网络；
- 身份暴露程度；
- 阵营变化；
- 当前任务目标。

5. 伏笔生命周期管理不完整
虽然当前已经有 foreshadowings 表，并支持 open / paid_off 等状态，但仍缺少完整生命周期设计，例如：
- hinted
- deepened
- abandoned

同时也缺少“建议何时回收伏笔”的主动分析能力。

6. 世界观更新机制还偏简单
当前世界观更适合初始化生成，后续动态更新世界规则的能力还需要完善。比如：
- 新地图出现；
- 新势力加入；
- 世界规则例外被揭示；
- 稀有能力体系出现。

这些内容目前缺少细粒度增量更新机制。

7. 前端尚未实现
当前交付的是后端骨架，尚未配套前端页面。也就是说：
- 暂无项目管理页面；
- 暂无角色卡页面；
- 暂无章节编辑器；
- 暂无伏笔表、时间线可视化；
- 暂无 AI 辅助工作台。

当前仍然主要依赖 Swagger 或 Postman 进行接口测试。

8. 审校能力仍较弱
当前审校更多是 Prompt 级分析，尚未形成完整的：
- 人设一致性检查；
- 世界观冲突检查；
- 节奏评分；
- 钩子强度分析；
- 网文爽点密度分析。

9. 缺少异步任务与队列机制
目前章节生成、世界观生成、大纲生成等接口大多是同步请求。
后续如果文本变长、模型耗时增加，建议引入：
- Celery / RQ / Dramatiq；
- 异步任务状态表；
- 前端轮询任务进度。

10. 缺少权限、用户体系、多租户支持
当前项目更像单人本地开发版本，尚未加入：
- 用户登录；
- 项目归属关系；
- 数据隔离；
- 接口鉴权；
- 多用户并发使用场景。

七、总结

当前 Novel Agent 项目已经完成了一套较为完整的后端骨架设计，并实现了项目管理、立项生成、世界观生成、角色生成、大纲生成、章节生成、记忆更新、缓存接入和向量检索等关键能力。

它已经不是一个简单的“调用大模型写一段文本”的 Demo，而是一个具备 Agent 工作流雏形的 AI 小说创作后端系统。

当前最突出的价值在于：
1）结构化工程已经搭好；
2）核心主链路已经跑通；
3）MySQL + Redis + Milvus 的分层存储架构已经具备；
4）后续继续扩展 Prompt、Memory、Tool 和前端界面，都有明确基础。

但与此同时，这个项目仍处于“第一阶段可运行骨架版本”，还需要继续补足：
- 稳定性；
- 检索增强质量；
- 角色/伏笔/时间线管理深度；
- 审校能力；
- 前端工作台；
- 用户与权限体系。

如果继续推进，建议优先围绕“稳定主链路 + 强化章节生成质量 + 完善记忆系统 + 建设前端工作台”这四条主线展开。
